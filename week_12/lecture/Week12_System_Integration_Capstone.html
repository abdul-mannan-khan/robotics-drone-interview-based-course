1. What is System Integration?

System integration is the process of combining individually developed and tested subsystems into a single coherent system. In autonomous drone engineering this means connecting perception, estimation, planning and control into a pipeline that operates reliably in real time.

Why Integration is the Hardest Part

The Integration Problem: Individual components that work perfectly in isolation often fail when combined. Timing mismatches, frame transform errors, inconsistent assumptions, and cascading failures make integration the most challenging phase of any robotics project.

Component Development

  • Well-defined inputs/outputs
  • Controlled test environment
  • Single failure mode at a time
  • Easier to debug

System Integration

  • Emergent behaviours
  • Timing dependencies
  • Cascading failures
  • Non-reproducible bugs

The Integration Gap

Research shows that integration and testing typically consume 40-60% of total project time in robotics. The "last 10%" of making a system work reliably often takes as long as the first 90%.

4. Integration Challenges

4.1 Timing and Synchronisation

Different sensors run at different rates: LiDAR at 10 Hz, IMU at 200 Hz, camera at 30 Hz. The controller expects consistent state estimates regardless of sensor availability.

Common Timing Failure: The planner receives a stale state estimate because the EKF has not yet processed the latest sensor data. The planned path is based on where the drone was, not where it is, causing oscillations or collisions.
\[ \tau_{\text{total}} = \tau_{\text{perception}} + \tau_{\text{estimation}} + \tau_{\text{planning}} + \tau_{\text{control}} \]

where:

  • \(\tau_{\text{total}}\) is the end-to-end pipeline latency (seconds)
  • \(\tau_{\text{perception}}\) is the time to process raw sensor data
  • \(\tau_{\text{estimation}}\) is the state estimation computation time
  • \(\tau_{\text{planning}}\) is the path/trajectory planning time
  • \(\tau_{\text{control}}\) is the control computation and actuation time

4.2 Frame Transforms

Every sensor has its own coordinate frame. The LiDAR frame, camera frame, body frame, and world frame must all be correctly related through the TF2 transform tree.

Frame Transform Bug: The camera obstacle detection reports positions in the camera frame, but the planner expects them in the world frame. Without the correct transform, obstacles appear in the wrong location.

4.3 Data Flow and Bandwidth

Bandwidth Management Strategies

  • Downsample point clouds before publishing (voxel grid filter)
  • Use compressed image transport
  • Publish processed results not raw data when possible
  • Use QoS profiles: BEST_EFFORT for sensors, RELIABLE for commands

4.4 Error Propagation

Errors in one subsystem cascade through the pipeline:

\[ e_{\text{tracking}} = e_{\text{control}} + e_{\text{trajectory}} + e_{\text{state}} + e_{\text{sensor}} \]

where:

  • \(e_{\text{tracking}}\) is the total tracking error observed
  • \(e_{\text{control}}\) is error from controller imperfections
  • \(e_{\text{trajectory}}\) is error from trajectory generation
  • \(e_{\text{state}}\) is error from state estimation inaccuracy
  • \(e_{\text{sensor}}\) is the raw sensor measurement error

5.2 Integration Testing

Test pairs of connected components. Verify that the planner produces trajectories the controller can track, or that the EKF correctly fuses IMU and GPS data.

5.3 Simulation Testing (SITL)

Run the complete system in Gazebo or AirSim. Test dangerous scenarios safely and repeatably.

5.4 Hardware-in-the-Loop (HIL)

Connect flight controller hardware to simulation. PX4 runs on real hardware but receives simulated sensor data, validating real-time performance.

5.5 Field Testing

Final validation on real hardware. Always follow safety procedures and have manual override ready.

6. Simulation-to-Reality Transfer (Sim2Real)

The Domain Gap

Figure 7: The simulation-to-reality domain gap

Bridging the Gap

ChallengeSimulationReality
Sensor noiseGaussian, zero-meanNon-Gaussian, biased, correlated
DynamicsRigid body, no flexVibration, prop wash, flex
LatencyDeterministic, lowVariable, jitter, buffering
EnvironmentStatic, texturedDynamic, varying light
CommunicationPerfect, instantPacket loss, delays

Domain Randomisation

Randomise simulation parameters during training so the system learns robustness:

  • Visual: Randomise textures, lighting, colours
  • Physical: Randomise mass, inertia, drag coefficients
  • Sensor: Randomise noise levels, biases, dropout rates
  • Environmental: Randomise wind, obstacles, starting positions

Transfer Learning Approach

Train in simulation with domain randomisation, then fine-tune with a small amount of real-world data. More sample-efficient than training from scratch on real hardware.

7. System Performance Metrics

7.1 End-to-End Latency

The total time from sensor measurement to actuator command must be less than the time to reach the nearest obstacle:

\[ \tau_{\text{safe}} < \frac{d_{\min}}{v_{\max}} \]

where:

  • \(\tau_{\text{safe}}\) is the maximum allowable end-to-end latency (s)
  • \(d_{\min}\) is the minimum obstacle detection distance (m)
  • \(v_{\max}\) is the maximum drone velocity (m/s)

7.2 Tracking Accuracy

\[ \text{RMSE} = \sqrt{\frac{1}{N} \sum_{i=1}^{N} \| \mathbf{p}_i^{\text{actual}} - \mathbf{p}_i^{\text{desired}} \|^2} \]

where:

  • \(N\) is the number of time samples
  • \(\mathbf{p}_i^{\text{actual}}\) is the actual position at step \(i\)
  • \(\mathbf{p}_i^{\text{desired}}\) is the desired position at step \(i\)

7.3 System Reliability

For \(n\) independent series components:

\[ R_{\text{system}} = \prod_{i=1}^{n} R_i \]

where:

  • \(R_{\text{system}}\) is the overall system reliability
  • \(R_i\) is the reliability of component \(i\)

Example:

5 subsystems at 99% each: \(R = 0.99^5 = 0.951\) — only 95.1% reliable!

7.4 Mean Time Between Failures

\[ \text{MTBF} = \frac{\sum_{i=1}^{n} T_i}{n_f} \]

where \(T_i\) is operating time per period and \(n_f\) is total failures.

12. Common Integration Failures

Case Study 1: The Phantom Obstacle

Symptom: Drone repeatedly stops and replans in clear space.
Root Cause: Obstacle detector used camera frame but occupancy grid expected world frame.
Fix: Apply tf2 transform from camera_optical_frame to map frame.
Lesson: Always verify frame transforms at every interface boundary.

Case Study 2: The Oscillating Drone

Symptom: Drone oscillates around waypoints.
Root Cause: EKF had 50 ms latency; controller assumed zero latency, always overcorrecting.
Fix: Compensate for latency or use predicted state at current time.
Lesson: Measure and account for pipeline latency.

Case Study 3: The Planning Timeout

Symptom: Drone hovers indefinitely after obstacle detection.
Root Cause: A* on 1 cm resolution grid took 5+ seconds.
Fix: Hierarchical planning: coarse grid global, fine grid local.
Lesson: Profile planners and set appropriate resolution.

Case Study 4: The Silent Sensor

Symptom: Works in sim, crashes on real hardware.
Root Cause: LiDAR driver drops messages; EKF drifts without corrections.
Fix: Sensor health monitoring; hover on 500 ms timeout.
Lesson: Implement timeout and fallback for every sensor input.

13.3 Health Monitoring

class HealthMonitor:
    def __init__(self):
        self.components = {
            'ekf': {'timeout': 0.1, 'last_seen': 0},
            'planner': {'timeout': 1.0, 'last_seen': 0},
            'controller': {'timeout': 0.02, 'last_seen': 0},
            'lidar': {'timeout': 0.2, 'last_seen': 0},
        }

    def check_health(self, current_time):
        status = {}
        for name, comp in self.components.items():
            elapsed = current_time - comp['last_seen']
            if elapsed > comp['timeout']:
                status[name] = 'CRITICAL'
            elif elapsed > comp['timeout'] * 0.8:
                status[name] = 'WARNING'
            else:
                status[name] = 'OK'
        return status
